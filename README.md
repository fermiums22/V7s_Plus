# V7s Plus mobile assistant base

Этот проект не про восстановление робота-пылесоса как пылесоса.

Главная идея: сделать домашнюю мобильную платформу для голосового/камерного ассистента. То есть не просто Алиса для лампочек и не только серверный Home Assistant, а физическая сущность, которая может ездить по дому, слышать/видеть через надстройку, говорить через динамик, возвращаться на зарядку и быть интерфейсом к домашнему ассистенту.

## Зачем здесь V7s Plus

У Виктора уже есть сервер с Proxmox, Home Assistant и TrueNAS. Серверная часть может тянуть распознавание речи, LLM/API, TTS, сценарии умного дома и хранение данных. Но сервер сам по дому не ездит.

Старый робот-пылесос V7s Plus выбран как готовая механическая и силовая база:

- аккумулятор и силовая разводка уже есть;
- есть два мотор-колеса с энкодерами;
- есть док-станция/зарядные контакты, значит в будущем возможна самозарядка;
- есть место и питание для дополнительной электроники;
- родную STM32-плату можно использовать как низкоуровневый контроллер движения.

Идеальная будущая архитектура:

```text
Home Assistant / сервер / OpenAI / TTS / STT
        ^
        | Wi-Fi / UART / другой канал
        v
верхняя "умная" надстройка: K210/камера/микрофон/динамик/сеть
        ^
        | команды движения / телеметрия
        v
STM32 плата V7s Plus: питание, моторы, энкодеры, датчики, док-зарядка
```

Сейчас мы работаем именно над нижним уровнем: заставить штатную STM32-плату надежно питать датчики, управлять колесами, считать энкодеры и стать управляемой базой для будущего ассистента.

## Важное правило проекта

Конфигурацию STM32 править через `V7s_Plus.ioc` и регенерить CubeMX.

Не править руками generated init-код в `Core/Src/main.c` вне `USER CODE` блоков. Пользовательскую логику держать в `USER CODE` или отдельных файлах, например `Core/Src/motor_control.c`.

Полезные команды:

```bat
flash.bat
reset.bat
rest.bat
```

`flash.bat` делает тихую инкрементальную сборку Debug, прошивает через ST-LINK и пишет подробные логи в `Debug/flash_build.log` и `Debug/flash_program.log`.

## Текущее состояние железа

MCU: `STM32F071V8Tx`.

Родная прошивка потеряна после RDP unlock, поэтому пишем свою минимальную прошивку.

Питание:

- батарея: 14.4 V Li-ion;
- `U7` TPS54331-like buck делает 5 V;
- `U8` делает 3.3 V;
- `SWITCHED_SENSOR_5V` это общая 5 V линия энкодеров/части аналоговой обвязки;
- `SWITCHED_SENSOR_5V` включает `PE5 -> Q21 -> Q20`;
- `Q24/PA8` не считать подтвержденным ключом питания энкодеров.

Колеса:

- левое колесо: `J5`;
- правое колесо: `J14`;
- колесо 70 mm диаметром;
- база между центрами колес 240 mm.

## Найденные пины моторов

Левый драйвер `U2`:

```text
PD8  -> U2 pin 1  nFAULT
PB7  -> U2 pin 3  PHASE
PC8  -> U2 pin 6  ENABLE / TIM3_CH3 PWM
PB1  -> U2 pin 15 VPROPI / ADC_IN9
PD3  -> левый энкодер
```

Правый драйвер `U6`:

```text
PD14 -> U6 pin 1  nFAULT
PE13 -> U6 pin 3  PHASE
PC6  -> U6 pin 6  ENABLE / TIM3_CH1 PWM
PC5  -> U6 pin 15 VPROPI / ADC_IN15
PE8  -> правый энкодер
```

Важно: `PE3` не прямой правый энкодер. Это смешанный сигнал через большие RC-цепи от `J16` и энкодерных/аналоговых узлов. Его не использовать как wheel encoder input, пока отдельно не разберем назначение.

## Что уже работает

В прошивке есть модуль:

```text
Core/Inc/motor_control.h
Core/Src/motor_control.c
```

Он уже умеет:

- включать `SWITCHED_SENSOR_5V`;
- стартовать PWM для левого и правого драйвера;
- задавать направление;
- считать левый и правый одноканальные энкодеры;
- вести позицию и скорость в encoder edges;
- пересчитывать колесо в обороты, метры и m/s;
- задавать движение колеса в оборотах;
- считать путь колеса для будущего разворота робота.

Текущий энкодерный коэффициент:

```c
MOTOR_L_ENCODER_EDGES_PER_WHEEL_REV = 1130.0f
```

Пока он общий/рабочий. Подобран на глаз и должен уточняться измерениями.

Геометрия:

```c
MOTOR_WHEEL_DIAMETER_M = 0.070f
MOTOR_WHEEL_BASE_M     = 0.240f
```

Последний проверенный двухколесный тест:

- оба колеса прошли вперед и назад;
- оба энкодера считались;
- правый `PHASE` инвертирован в коде, чтобы логический forward совпадал с левым колесом;
- после теста позиции были около нуля.

## Текущая прошивка

На момент этого README в `main.c` стоит тестовый сценарий двух колес:

```text
оба колеса +2.0 оборота
пауза
оба колеса -2.0 оборота
```

Это тестовый bring-up код, а не финальное поведение робота.

Следующий нормальный шаг: убрать тест из старта и сделать явный командный интерфейс/режимы, чтобы внешний контроллер или отладчик мог задавать:

- стоп;
- скорость левого/правого колеса;
- проехать дистанцию;
- повернуть на угол;
- вернуть телеметрию.

## Ближайшие цели

1. Привести `motor_control` к нормальному двухколесному API:
   - `left/right speed`;
   - `linear speed`;
   - `angular speed`;
   - `drive distance`;
   - `turn angle`.

2. Сделать устойчивый регулятор скорости:
   - сейчас контуры грубые;
   - нужно отдельно калибровать левый и правый encoder edges/rev;
   - учесть разный знак направления колес.

3. Подготовить связь с верхней надстройкой:
   - UART/I2C/SPI или другой простой протокол;
   - команды движения;
   - телеметрия батареи/ошибок/энкодеров;
   - emergency stop.

4. Разобрать оставшиеся датчики и питание:
   - док-станция и зарядка;
   - батарейный мониторинг;
   - датчики препятствий/пола;
   - что делает `PE3`;
   - что делает `U10`.

5. Потом уже строить верхний уровень:
   - микрофон/камера/динамик;
   - Wi-Fi модуль или K210/другая плата;
   - связь с Home Assistant/сервером;
   - голосовой ассистент через STT/LLM/TTS.
